For a long time, I’ve been skeptical of malleable software.

There has always been this HyperCard 2.0 dream behind it. The idea that users should be able to shape their own tools, define their own workflows, and gradually turn software into something uniquely theirs. It is an appealing vision. It treats software as a medium and the user as a creator.

I understand the appeal. I have just never found it very believable. Most people do not want or have time, or capacity to build their own tools. They want something they can rely on, share with others, and return to later without having to reconstruct how it was put together. Tools are for doing the work, not for constantly tuning the tool itself. The tool building is a side quest, not the main driver of any real work. No matter how many ways you paint and shape the hammer, your day to day job is still to hit the nail.

Why I am thinking about this now that agents introduce a new kind of flexibility. Instead of asking users to craft their tools, the system can interpret intent and adapt to it. You describe what you want, and the agent will deal with the complexity of getting it done.

You agents are the closest thing we have to the HyperCard dream and can certainly be used that way, the reality is that you don’t actually need it.

Agents are already almost infinitely malleable on their own and offer the variety of needs you might have, or you just use them to build opinionated software which then you adjust as you go.

The idea of malleable, or customizable software kind of loses its purpose. You don’t need a complicated middleware system of malleable blocks or ways to customize. You need a system and agent to run it.

Agent-driven software anyways suggests flexibility should be computed at runtime with the user intent. The system can respond to variation without requiring every variation to become a lasting part of the software design. Each model, harness and capability will evolve this even more in the future.

There is also a cost here that becomes more obvious with agents.

Malleable systems expand the amount of information the agent has to understand before it can do useful work. Every configurable layer, custom workflow, and user-defined structure adds more context to parse, more exceptions to reason through, and more ambiguity to resolve. That cost shows up directly in tokens, latency, inference, and failure rates. The agent spends more of its budget figuring out how the system works, and less of it doing the thing it was asked to do.

This is not that different from humans. A person working inside a highly customized system also spends more time decoding the setup before they can act. Agents face the same problem, except their cost is computational instead of cognitive.

That is why I think strong system structure matters more in the agent era, not less. The best systems will be more opinionated at the core. They will define clear objects, clear workflows, and clear expectations. On top of that, agents provide adaptability and flexibility within it.. They handle variation of workflows without requiring users to formalize it into structure.

What matters more in this world is not how much the interface can be reshaped. It is whether the system is constructed well to efficiently feed the right context into agent execution leading to the desired outcome.

For a long time, malleable software was framed as the more empowering vision because it gave users more control. I increasingly think the more humane vision is the opposite.

Software and agents should carry more of the burden. Users should get better outcomes without having to become architects of their own tools.

One might argue that agents are perfect examples of malleable software, it adjusts to users needs. Fine, but I’d still call them agents and let the malleable software be rest in peace.

The real promise of this era is not that everyone shapes and customizes their own tools with malleable systems.

It is that agents become capable enough that we no longer need to.
